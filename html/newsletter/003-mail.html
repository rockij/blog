React, Vue 프로젝트 환경에서 퍼블리셔의 역할은?

요즘 React나 Vue 기반 프로젝트를 주로 진행하면서 퍼블리셔의 역할에 대해 스스로 고민하게 됩니다.

예전처럼 단순히 마크업을 하고 CSS와 JS를 활용해 인터랙션을 구현하던 시기에도 개발자와 어디까지 작업을 나눌지에 대한 이슈가 있었습니다. 하지만 이러한 프레임워크 환경에서는 역할의 범위가 더 다양해지고 난이도도 높아졌다고 볼 수 있습니다.

지금은 퍼블리싱을 할 때 개발이 바로 이어질 수 있는 UI 구조를 먼저 만드는 것에 대해 더 고민해야 합니다.
조금 더 구체적으로 이야기하자면 디자인이 나오면 기본 레이아웃 작업을 진행하고, header, content, footer 형태의 레이아웃 컴포넌트를 구성합니다.
이후 버튼, 아코디언, 탭과 같은 공통 UI 컴포넌트를 먼저 구현해 개발 단계로 넘기는 방식으로 작업합니다.
이러한 방식은 개발자가 비즈니스 로직에 집중할 수 있게 해주기 때문에 프로젝트 진행 측면에서도 효율적이라고 생각합니다.

각 컴포넌트는 기본 스타일을 정의하고, 실제 화면에서 사용하는 방법까지 고려해 props와 event를 주고받는 구조로 설계합니다.
또한 해당 컴포넌트를 사용하는 화면 예시까지 포함해 전달하고, 이후 개발이 완료된 컴포넌트는 개발자와 함께 공용으로 관리하는 방식이 좋다고 생각합니다.

컴포넌트 관리 역시 퍼블리셔와 개발자가 각각 따로 관리하는 것이 아니라 하나의 컴포넌트를 공용으로 관리하는 구조가 업무 효율 측면에서 더 바람직합니다.
다만 프로젝트에 따라 공통 개발자가 없거나 컴포넌트 관리 자체의 필요성을 크게 느끼지 못하는 경우도 있습니다. 이런 경우에는 프로젝트 후반으로 갈수록 전체적인 UI 품질이 떨어지는 상황을 경험하기도 했습니다.
따라서 이러한 상황에서는 컴포넌트 관리의 필요성에 대해 퍼블리셔 측에서도 적극적으로 제안할 필요가 있다고 생각합니다.

React, Vue 기반 프로젝트에서는 컴포넌트 작업의 중요도가 매우 높습니다. 이런 환경에서 퍼블리셔의 역량을 확장하고자 한다면 초기 단계부터 컴포넌트 설계에 깊이 관여하고, 구조와 방향성에 대해 먼저 제안할 수 있어야 한다고 생각합니다.

또한 퍼블리셔 그룹의 작업 현황판을 만들어 관리해야 하는 상황이라면, 직접 router 설정을 구성하는 것 역시 퍼블리셔가 충분히 수행할 수 있는 영역이라고 생각합니다.

결국 퍼블리셔가 어디까지 작업할 수 있는지에 대해 개발팀과 사전에 협의하고 정리하는 과정이 필요합니다.
이러한 기준이 명확하지 않으면 업무 범위가 모호해지거나 협업 과정에서 커뮤니케이션 문제가 발생할 수 있기 때문입니다.
작은 기준의 차이가 결과를 바꿉니다. 다음 뉴스레터에서 다시 뵙겠습니다.
